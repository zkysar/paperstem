Auto-section classification — UX mockup v4

Sections appear on the DAW timeline while the project is still a local draft — before the user clicks Save. The audio is already loaded locally; classification runs on that audio. The user reviews, renames, deletes, and only then commits to the server.

The flow: "+ New project" → pick folder → audio loads locally → DAW renders → classification runs on the local audio → sections appear in the lane → user reviews → click Save on the draft banner → upload + commit. Sections are persisted along with the project create, so the saved state matches what the user saw in draft.
Flow A

Picking a folder → draft mode

The user clicks "+ New project" and picks a folder of stems. Paperstem enters draft mode: activeProjectId is null, the audio is loaded from local blob URLs (no upload yet), the DAW renders the audio for review, and a thin orange banner indicates the project hasn't been committed. The Save button lives on this banner.

Auto-classification runs here. The audio is already decoded in the browser; we feed it to YAMNet + chroma the moment the DAW finishes loading. By the time the user has reviewed a few seconds of audio, sections start appearing in the section lane.

Flow B

Draft DAW — detection in progress

Right after the user picks a folder. The DAW shows the local audio (no upload yet). Draft banner reminds them the project isn't saved. Detection banner reports progress; sections will appear in the lane as they're identified. The user can play, mute, navigate around while this runs.

Decisions:
  • Save button on the draft banner is disabled (greyed) while detection is running. Tooltip: "Detecting sections…" Or we could enable Save anytime and have sections POST after upload — see Flow C decisions.
  • Detection banner is non-blocking — the user can already play, mute, add manual sections, etc. while it runs.
  • If detection fails, the detection banner becomes "Detection failed (try again)" with a retry link. Save still works without sections.
  • If the user creates a manual section in this window, the auto-detector respects it: any auto section that would overlap gets dropped before being written.
  • "Discard" on the draft banner is the existing Paperstem behavior — abandons the local draft, no upload happens.
Flow C

Draft DAW — sections detected, Save now enabled

Detection completes, sections fade into the lane, Save button on the draft banner enables. The user can rename, delete, or just hit Save. Whatever the user accepts (by not deleting) gets committed when they Save — the saved state matches what they see now.

Two visual variants below. Functional behavior identical; only the treatment differs.

The two-variant question:
  • Variant 1 is the radical simplification. Treat sections like first-class citizens regardless of origin. The user sees them, edits them, life is simple. Risk: someone opens the project later and doesn't realize the system made these — might be surprised.
  • Variant 2 keeps a subtle "not yet vouched-for" marker. Touching the section (any edit, or just opening the popover) removes the dashed treatment and the dot. Slightly more state to manage; clearer mental model.
  • My recommendation: Variant 2, but only for the very first viewing of a project — once the user has seen them, treat as normal sections forever after. Even untouched ones lose the "fresh" look on the second visit.
Flow C-post

After Save — same DAW, draft banner gone

User clicks "Save to Paper Straw" on the draft banner. The drawer (existing UploadDrawer with project name / date fields) opens for confirmation, then the upload happens. Sections POST alongside the project create. When everything's done, the DAW transitions to "saved": the draft banner disappears; sections remain (still with the Variant 2 "fresh" treatment until the user touches each one).

Notes:
  • What persists across the Save boundary: the sections (with their current names + any user edits), the fact that they came from auto-detection, the confidence scores.
  • What disappears: the draft banner, "draft" status text.
  • If the user reloads later, the same sections still have the "fresh" dot until they're touched (since the schema stores source='auto'). Once touched, they convert to source='manual' and look like any other section.
  • Open question: does the user need a way to "accept all" sections at once (bulk-clear the fresh treatment)? Probably not — the treatment is subtle enough that it shouldn't bug them; ignoring it is fine. If we ever add a "review unreviewed" gesture, it can come later.
Flow D

First-time toast — "Hey, we did this"

The first time a user sees auto-detected sections, a small toast explains what happened. Appears in draft mode, before the user has hit Save. Auto-dismisses on interaction. After they've seen it once on any project, no more toast.

5 sections detected. Edit names or delete any you don't want. We listened to the audio and grouped it into songs, chatter, and tuning.
Flow E

Clicking a section — same popover as today

No special "Accept" action. Click → rename, delete, move boundary — identical to what manual sections get today. If you're using Variant 2, any of these interactions removes the "fresh" treatment.

Flow F

CLI imports — same DAW, no separate review path

Tascam Model 12 imports run classification on the user's machine and POST the sections. When the user opens the project in the web app for the first time, the DAW renders the same way it would for a fresh web import: with the "fresh" treatment (Variant 2) until they touch each section. No project-row badge, no separate "review" screen.

What changed from v1/v2:
  • No badge on the project row. When the user opens the project, the DAW itself communicates "here's what the system suggested" — same as for web imports.
  • One UI for two import paths. Web and CLI converge on the DAW as the only review surface.
