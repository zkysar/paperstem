Auto-section classification — UX mockup v2

The flow: classification happens inline in the upload drawer the moment the user picks a directory. The user reviews proposed sections right there — rename or discard — and then commits everything in one Save. No separate "review later" step on the project page.

What collapsed from v1: the opt-in checkbox is gone (it just always runs); the project-page "auto vs manual" visual distinction is gone (sections are already approved at commit time); the "Accept all confident / Discard all" toolbar is gone (you do that inline in the drawer); the "re-classify" button is gone (re-import to re-classify). The badge on project rows still exists — but only for CLI-imported projects, since the CLI has no interactive review step.
Flow A

Picking the directory — classification starts immediately

Right after the user picks files (or drops a folder), the drawer opens and YAMNet + chroma start running on the full mix. While that happens, the user can name the project and set the recorded date. Both happen in parallel — the user's metadata input fills the otherwise-wasted 10–15s of classification compute.

Decisions in this mock:
  • No "auto" badging in the drawer. Every row is just "a section the system found." Once you Save, they're committed as ordinary sections.
  • Type pills (music / chatter / tuning) replace the auto-vs-manual visual distinction. They convey what kind of segment it is, not where it came from.
  • Tentative matches (Tennessee Stud?) show the song name with a question mark + italic + small "low" tag. Still committed as a manual section if you Save without editing — but visually flagged so you notice them.
  • "Skip detection and save now" is the escape hatch — if the user knows detection is wrong / running long, they can bypass and save without sections.
  • Save is disabled until detection finishes — keeps the model simple. (Alternative: enable Save anytime; detection results POST after upload completes.)
  • Mini timeline at the top gives proportional visual context. The list below is the workspace for renaming/discarding.

Interactions — rename, discard, restore

Interaction details:
  • Click anywhere in the name field to rename. Tab cycles between rows for fast keyboard editing.
  • Discard doesn't remove the row — it strikes through and offers restore. Easy to undo before commit; goes away on Save.
  • The mini timeline reflects discarded sections too (greyed-out), so you can see the resulting structure as you edit.
  • Adjusting section *boundaries* in the drawer is intentionally not supported. Boundary drag is a precision task that belongs in the full project view (zoomed timeline). The drawer is for naming and pruning. If the user wants to fix a boundary, they save first, then adjust in the timeline.
  • What about adding a new section the model missed? Same answer — defer to the project view. The drawer is the "review what the model found" view, not the "compose from scratch" view.
Flow C

Save — uploading + committing in one shot

The user clicks Save. The drawer disables its inputs and shows the upload progressing. Sections are POSTed alongside the project create request (or right after), so when the drawer closes the project page already has them.

Notes:
  • Sections render collapsed (mini timeline only, no row list) once committed — the workspace is for editing; once you've hit Save you're just watching.
  • If the upload fails partway, the user can retry; sections are held client-side until the project create succeeds.
  • If sections are large (e.g., many chroma fingerprints for a long practice), POSTing them separately after the project create avoids inflating the upload request.
Flow D

CLI imports — the only "review later" path

For Tascam Model 12 imports via bin/import-from-device.ts --auto-classify, there's no interactive drawer to review in. The CLI runs classification, POSTs the sections, and the user reviews next time they open the web app. A slate badge on the project row tells them which projects have un-reviewed auto sections.

How review works for CLI imports:
  • The project page shows the sections as provisional — same dashed-border + slate visual treatment from the v1 mockup, since these weren't reviewed at import time.
  • The user accepts or discards individual sections in the project view; once all are accepted (or all discarded), the badge clears.
  • This is the only place in v2 where the "auto vs manual" visual distinction lives. The web import path doesn't need it.
